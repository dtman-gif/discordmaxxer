# v0.2 — Tier Roster Sync + Subscription Model

> Status: design — not yet user-facing. Phase-1 scaffold (`data/tiers.json` + `plugins/_dm-shared/roster.ts`) lands ahead of v0.1.0 stable as preparatory infrastructure. Full subscription / payment integration deferred until pricing infra is in place.

## Problem

The v0.1 grant system (`DiscordmaxxerGrant`) stores tier grants **locally** in the user's Vencord settings. When User A grants User B `MAXXER+`, only A's client treats B as `MAXXER+` — B's own install still resolves itself as `FREE` and stays locked into mandatory presence + brand-pinned defaults. There is no shared state between two Discordmaxxer instances.

For tiers to *do anything for the recipient*, we need a single source-of-truth roster every install can read.

### tiers.json shape

```jsonc
{
    "version": 1,
    "issuedAt": "2026-05-08T00:00:00Z",
    "users": {
        "<user-id>": {
            "tier": 3,                            // Tier enum: 0..3
            "via": "owner" | "subscription" | "grant" | "comp",
            "grantedAt": "2026-05-01T00:00:00Z",
            "expiresAt": "2026-06-01T00:00:00Z",  // nullable for "permanent"
            "grantedBy": "<user-id>"              // optional, for friend-grants
        }
    }
}
```

Client semantics:
- `version` — bump on shape changes; client refuses to parse newer versions than it knows.
- `issuedAt` — diagnostic only.
- `expiresAt: null` — never expires (owner / lifetime / comped).
- `expiresAt < now()` — entry ignored (effectively FREE).
- Within 7 days of `expiresAt` — client surfaces a "renewing soon" hint in the VIP card and grant menu.
- 24h grace period after `expiresAt` — UX-only soft window so a brief subscription gap doesn't immediately strip features mid-session.

## Subscription model

**The mechanic that justifies recurring revenue is the friend-grant budget**, not raw cosmetic value. Subscribers don't pay for features alone — they pay for the ability to share them. Each active subscription becomes a marketing engine.

### Pricing (proposed)

| Tier | Monthly | Annual | Friend-grants budget /month | Grant max-tier | Grant duration |
|---|---:|---:|---:|---|---|
| FREE | $0 | $0 | — | — | — |
| MAXXER | $4 | $40 (-16%) | 1 | MAXXER | 7 days |
| MAXXER+ | $7 | $70 (-16%) | 3 | MAXXER | 30 days |
| MAXXER++ | $12 | $120 (-16%) | 5 | MAXXER+ | 30 days |

### Why grants drive renewal

1. **Sharing tax.** A MAXXER+ user who's never granted a friend has paid $7/mo for cosmetic perks they could find free elsewhere. The grants budget is the asymmetric value: $7 to gift up to 3 friends 30 days of MAXXER each, every month.
2. **Conversion funnel.** Granted friends experience paid features for 7-30 days. The data Hypixel saw with their MVP++ subscription: ~12% of granted users converted to their own paid sub.
3. **Skip-month penalty.** Unused grants don't roll over. If you skip a month, the granted friend's tier expires and the grants pool resets — meaning your effective $/grant cost rises if you don't share. Encourages monthly active engagement.
4. **Status flex.** The grant action surfaces as a Discord toast on the recipient's side: "Diggy granted you MAXXER for 30 days." Recipients see who has the budget to spend.

### Recurring perks beyond grants

- **Seasonal theme drops.** New monthly/quarterly limited-edition theme. Subscribers receive automatically; lapsed users keep what they had but stop receiving new drops. (Battle-pass-style, drives both new sub and re-sub.)
- **Cloud-synced themes & settings** across devices. Recurring infra cost legitimately justifies recurring fee.
- **Tenure badge.** "MAXXER++ since May 2026" — visible to other Discordmaxxer users, signals durable commitment.
- **Beta features.** New plugins ship to MAXXER+ first; FREE/MAXXER users get them 2 weeks later. Drives re-up timing for early-access seekers.
- **Founders' Discord channel.** Maintained by the project, gated by valid subscription. Cancellation kicks the user (recoverable on resub).
- **Priority issue support** — subscriber issues get a label that gets answered first.

### Anti-pattern guardrails

- **No paywalled essentials.** The base client + Vencord plugin set + 5 themes + TournamentMode + CompactView + MassDelete all stay FREE. We don't downgrade existing free features into paid ones.
- **No artificial throttling of FREE users.** No "free tier loads slower" or rate-limited features. The subscription is purely additive.
- **No re-asserting account settings on resub.** Same anti-self-bot rule that governs `DiscordmaxxerBadge` channels B/C/D applies here. Subscription elevation = unlocks features locally, never PATCHes Discord settings on the user's behalf.

## Authoring flow (the "publish a grant" UX)

Day-1 (manual, owner-only):
1. Grant happens locally via the existing right-click menu (`DiscordmaxxerGrant`).
2. New "Publish to roster" action in the menu opens a pre-populated GitHub web-URL:
   `https://github.com/dtman-gif/discordmaxxer/edit/main/data/tiers.json` with the entry pasted in the URL fragment for paste-in.
3. Owner commits manually.

Day-30 (semi-automated):
1. Grant action POSTs to a Cloudflare Worker / Vercel function with the owner's auth token.
2. Worker validates auth, edits `tiers.json` in the repo via the GitHub Contents API, commits.
3. CDN serves the updated JSON within seconds.

Day-90 (subscription-integrated):
1. Stripe / Lemonsqueezy webhook → backend writes a tier entry on `checkout.session.completed`.
2. Subscription `customer.subscription.deleted` → backend nulls or expires the entry.
3. `customer.subscription.updated` (plan change) → backend updates tier.
4. Roster JSON is now derived state of the payments DB; backend re-emits on every change.

## Domain & hub integration

This sits inside the broader **maxxer suite** plan: 6 standalone projects (`dropmaxxer`, `aimmaxxer`, `clipmaxxer`, `optimizationmaxxer`, `viewmaxxer`, `editmacros`) eventually merged behind a shell site at a single domain.

**Roster URL plan as the hub matures:**
1. **Now (rc3 era):** `https://raw.githubusercontent.com/dtman-gif/discordmaxxer/main/data/tiers.json`
2. **Custom-domain era (you've bought one):** `https://<hub-domain>/discordmaxxer/tiers.json` — same JSON shape, served from the hub site. The roster URL is a single config constant in `roster.ts`; flip it when you're ready.
3. **Multi-product era:** `https://<hub-domain>/api/tiers/<product>.json` — the maxxer-suite-wide subscription system. A single subscription elevates the user across every product (`dropmaxxer`, `aimmaxxing`, etc). Each client reads its own product's roster, but the same backend serves them.

The unified-sub mechanic is what makes the hub site valuable: *one subscription, six products*. Discordmaxxer alone might not justify $12/mo to many users; the bundle absolutely does.

## Open questions

1. **Annual vs monthly default.** Default to monthly checkout (lower friction) or annual (higher LTV at signup)?
2. **Stripe vs Lemonsqueezy vs Paddle.** Lemonsqueezy handles VAT/sales-tax compliance for us — for a one-person shop, that's worth ~5% extra fees vs Stripe.
3. **Grant max-tier ceiling.** Should MAXXER++ be allowed to grant MAXXER++ to a friend (full-tier transfer for 30 days), or capped at MAXXER+ (one-tier-below-self)? Capping prevents a single user from making everyone they know MAXXER++ — preserves perceived rarity.
4. **Refund policy for unused grant budget.** A user who cancels mid-month has unused grants. Snapshot or void? Void is simpler, but a snapshot could be a re-sub hook ("you have 3 unused grants — re-up to use them").
5. **Permanent grants for genuine contributors.** Anyone who ships a meaningful PR / fixes a bug / writes a plugin gets MAXXER+ permanently. Hypixel's "MVP++ for life" for high-value community members is excellent retention storytelling.
